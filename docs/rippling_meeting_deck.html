AgentAuth: dynamic capability scoping + monitoring, not approve-everything

Today's model, per your own docs, is binary: the agent holds the requester's full inherited permission profile, and the only gate is a human clicking approve on every single write, "the same as if it were made by a human." That's exactly why every gap on the last three slides exists — the thing being manually approved wasn't cryptographically bound to what actually executes (currency, tenant, object id, connector, aggregate amount, Unicode-rendered text), so the human click was often approving something other than what ran.

Our core mechanism is the third option between "full standing permission" and "a human eyeballs every write": a signed, single-use capability token scoped to exactly one task — this tool, these exact arguments, this tenant, this expiry, nothing broader — plus continuous agent monitoring: every call logged into a tamper-evident, hash-chained receipt chain, so any deviation is either blocked outright or surfaced for review, not silently trusted.

That's what lets you reduce manual approval load instead of adding another layer of it:

  • Human attention on real risk, not every write — a token scoped to exact args/tenant/aggregate limits can safely auto-execute in-scope actions; approval clicks concentrate on what's genuinely novel or over-threshold
  • One capability rail across every connector — legacy/partner/MCP paths carry the same scoped-token guarantee as the primary write path, instead of each connector needing its own bespoke (or missing) trust story
  • What's approved is what executes — the token binds the exact rendered payload, byte for byte, so Unicode smuggling or silent object-id substitution can't slip between preview and execution
  • Every ambient signal becomes monitored, not authoritative — skill selection, incident-channel overrides, and memory precedent are visible in the trace but can never themselves satisfy the capability gate
